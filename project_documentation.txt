1. PROJECT OVERVIEW & VISION
--------------------------------------------------------------------------------

NEXUS is a sophisticated, real-time Load Balancing Simulator designed to provide 
visual clarity and educational insight into the world of distributed systems. 
At its core, it is a high-performance Python-based simulation engine that models 
the behavior of multiple server nodes, traffic patterns, and the complex 
algorithms used to distribute work across them.

In the modern era of cloud computing, load balancers are the unsung heroes of 
scalability. NEXUS brings these invisible giants to life with a high-fidelity 
hacker-themed dashboard that visualizes every request, every connection, and 
every millisecond of latency.

Key Capabilities:
- Visualize 6 different industrial-grade load balancing algorithms.
- Simulate server health, failures, and real-time recoveries.
- Monitor real-time metrics including Requests Per Second (RPS), Success Rates, 
  and Latency heatmaps.
- Experiment with Auto-Scaling and Raft-based consensus mechanisms.
- Experience a premium "Matrix-style" UI designed for maximum impact.

--------------------------------------------------------------------------------
2. THE STORY BEHIND NEXUS
--------------------------------------------------------------------------------

The genesis of NEXUS began with a simple question: "How do we actually 'see' 
traffic distribution?" Most production load balancers (like Nginx, HAProxy, 
or AWS ELB) are black boxes that output logs or dry metrics. For developers 
learning about system design, there was no intuitive way to witness the 
mathematical beauty of a Weighted Round Robin or the efficiency of 
Least Connections in a dynamic environment.

The Project Vision:
The vision was to create a "digital twin" of a production environment. Not just 
a toy, but a simulator that respects the physical constraints of computing—CPU 
cycles, memory overhead, and network jitter.

The Aesthetic Choice:
We chose the "Cyberpunk / Matrix" aesthetic not just for style, but because 
it represents the "under-the-hood" reality of the internet. The green-on-black 
visuals emphasize the raw data flowing through the system, making the 
simulation feel like a mission-control center for a global network.

3. TECH STACK DEEP DIVE
--------------------------------------------------------------------------------

NEXUS is built on a modern, decoupled stack that prioritizes speed, 
concurrency, and visual excellence.

BACKEND (The Brain):
- Python 3.10+: The primary language for logic and simulation.
- FastAPI: A high-performance web framework for the REST API and WebSocket 
  management. It provides the asynchronous foundation needed for high-frequency 
  updates.
- Pydantic: For robust data validation and serialization.
- Asyncio: Powering the non-blocking simulation loop and concurrent request 
  handling.
- Python-Dotenv: For secure environment configuration.

FRONTEND (The Nerve Center):
- Vanilla JavaScript (ES6+): Used for all logic to ensure zero-dependency 
  bloat and maximum browser performance.
- CSS3 (Vanilla): Custom design system using CSS Variables for the dynamic 
  Matrix theme, glow effects, and responsive layouts.
- Chart.js: Used for high-fidelity real-time analytics and metric 
  visualizations.
- WebSocket API: Providing the low-latency bridge between the simulation 
  engine and the UI.
- Lucide Icons / Google Fonts: For premium typography and iconography.

INFRASTRUCTURE & DEVOPS:
- Docker & Docker Compose: For containerization and easy environment 
  replication.
- Nginx: Used as a production reverse proxy and static file server.
- Vercel: The primary deployment target for the web-accessible version, 
  leveraging Vercel's serverless functions and edge network.

5. SYSTEM ARCHITECTURE: UNDER THE HOOD
--------------------------------------------------------------------------------

NEXUS operates as a decoupled system where the Backend "thinks" and the 
Frontend "shows".

1. The Simulation Engine (SimulationEngine):
   This is a Singleton class that manages the "World State". It maintains a 
   list of ServerNode objects and a MetricsCollector. It runs an async loop 
   (the "Tick Loop") that advances time in the simulation.

2. The Virtual Servers (ServerNode):
   Each server is not just a counter. It simulates CPU usage, memory pressure, 
   and connection overhead. If a server is overloaded, its response time 
   increases and its failure probability rises—exactly like a real server.

3. The WebSocket Bridge:
   Every 500ms (configurable), the Backend snapshots the entire simulation 
   state, serializes it to JSON, and broadcasts it to all connected clients. 
   This ensures the UI is always perfectly in sync with the simulation.

4. The API Layer:
   REST endpoints allow the user to intervene in the simulation—triggering 
   spikes, killing servers, or changing algorithms—without restarting the loop.

--------------------------------------------------------------------------------
6. THE SIMULATION ENGINE: TICK-BY-TICK LOGIC
--------------------------------------------------------------------------------

The "Tick" is the heartbeat of NEXUS.

What happens in a single 500ms Tick?
1. Request Generation: Depending on the "Traffic Mode" (e.g., Heavy = 8 
   req/tick), the engine generates a batch of SimulatedRequest objects.
2. Load Balancing: The current algorithm (e.g., Least Connections) is 
   queried to pick a destination for each request.
3. Execution: The request is "sent" to the server. The server calculates 
   a response time based on its current load and health.
4. Server Update: Each server processes its active connections, cooling 
   down its CPU and finishing old requests.
5. Metrics Aggregation: RPS, success/failure counts, and average latencies 
   are updated.
6. Alert Evaluation: The system checks if any thresholds (like 90% CPU) are 
   breached.
7. Broadcast: The state is pushed to the UI.

7. LOAD BALANCING ALGORITHMS: THE CORE MATHEMATICS
--------------------------------------------------------------------------------

NEXUS implements 6 industrial-standard algorithms:

1. ROUND ROBIN:
   The simplest form. It cycles through the list of healthy servers sequentially.
   Best for: Identical server hardware and predictable traffic.

2. WEIGHTED ROUND ROBIN:
   Similar to Round Robin but respects server "Weights". A server with weight 2 
   gets twice as many requests as a server with weight 1.
   Best for: Heterogeneous environments (mixing powerful and weak servers).

3. LEAST CONNECTIONS:
   A dynamic algorithm that routes traffic to the server with the fewest active 
   requests.
   Best for: Requests with varying processing times (e.g., small API calls vs 
   large report generations).

4. IP HASH:
   Uses the client's IP address to consistently map them to the same server. 
   This ensures "Session Persistence" (Sticky Sessions).
   Best for: Applications requiring local session state.

5. RANDOM CHOICE:
   Uniformly selects a server at random. Surprisingly effective at scale.
   Best for: Simple systems where distribution overhead should be minimal.

6. LEAST RESPONSE TIME:
   The most advanced. It tracks the moving average of response times and 
   prioritizes the fastest-responding node.
   Best for: Optimizing user experience in high-latency environments.

8. RAFT CONSENSUS: THE CONTROL PLANE SIMULATION
--------------------------------------------------------------------------------

NEXUS includes a simplified Raft Cluster simulation. In a real-world scenario, 
the Load Balancer itself must be highly available. 

- Leader Election: The simulator models nodes voting for a "Leader". If the 
  leader node fails, the "Control Plane" goes down briefly until a new 
  election completes.
- Traffic Impact: If the Raft cluster is in the middle of an election, the 
  load balancer cannot route traffic, simulating a "Split-Brain" or 
  "Control Plane Failure".

This adds a layer of depth, showing that the load balancer is also a 
distributed system with its own failure modes.

11. CHALLENGES, SOLUTIONS & PERFORMANCE OPTIMIZATIONS
--------------------------------------------------------------------------------

CHALLENGE: Real-time synchronization of 100+ variables every 500ms.
SOLUTION: Instead of sending every change, we use a "State Snapshot" pattern 
via WebSockets, reducing overhead and ensuring UI consistency.

CHALLENGE: Python's Global Interpreter Lock (GIL) and high-concurrency 
simulation.
SOLUTION: Leveraged Asyncio for non-blocking I/O and simulation logic, 
allowing the engine to handle thousands of "virtual requests" without 
blocking the main thread.

CHALLENGE: Keeping the UI smooth while updating charts at high frequency.
SOLUTION: Implemented "Request Batching" on the frontend. The UI only redraws 
on the WebSocket tick, preventing browser layout thrashing.

--------------------------------------------------------------------------------

NEXUS is designed for "One-Click Deployment".

DOCKER:
The project includes a multi-stage Dockerfile that builds a lightweight

NGINX:
Used as a reverse proxy to handle SSL termination, static file caching, 
and WebSocket connection upgrades.

VERCEL:
The project is optimized for Vercel. 
- The Frontend is served as static assets from the Edge.
- The Backend runs as a Vercel Serverless Function (Python).
- *Note*: Due to serverless execution limits, the WebSocket bridge is 
  automatically augmented with an adaptive polling fallback for 
  environments that do not support long-running sockets.

13. API REFERENCE & WEBSOCKET PROTOCOL
--------------------------------------------------------------------------------

NEXUS provides a full RESTful API for external control:

- POST /api/simulation/start: Begins the tick loop.
- POST /api/simulation/stop: Pauses the tick loop.
- POST /api/simulation/spike: Triggers a 5x traffic burst.
- GET  /api/metrics: Returns a JSON snapshot of current stats.
- POST /api/servers/add: Adds a new node to the pool.

WEBSOCKET DATA PACKET:
{
  "simulation": { "running": true, "algorithm": "Least Connections" },
  "servers": [...],
  "metrics": { "rps": 12.5, "latency_avg": 42, "success_rate": 99.8 },
  "alerts": [...]
}

14. USER MANUAL: HOW TO OPERATE THE SIMULATOR
--------------------------------------------------------------------------------

1. LAUNCH: Run `python run.py` and visit `localhost:8000`.
2. START: Click the "START" button to begin generating traffic.
3. EXPERIMENT: Switch between algorithms using the dropdown. Watch the 
   green request lines shift their patterns.
4. CHAOS: Use the "KILL" button on a server to simulate a crash. Watch how 
   the load balancer redistributes traffic to the survivors.
5. ANALYZE: Head to the "Analytics" tab to see long-term performance 
   graphs and "Cost/Efficiency" ratings for each algorithm.
